# AGENTS.md

## Project Overview

A production-ready TypeScript library starter template. Uses Bun as runtime/package manager, Biome for lint+format (no ESLint/Prettier), tsdown for building (Rolldown-based), lefthook for git hooks, changesets for versioning.

## Setup Commands

```bash
bun install              # install dependencies
bun install -d <pkg>     # add dev dependency
bun install <pkg>        # add runtime dependency
bun install --frozen-lockfile  # CI install (exact versions)
```

## Development Workflow

```bash
bun run dev               # tsdown --watch (rebuild on file change)
bun run typecheck         # tsc --noEmit (type-check src/ + test/)
bun run lint              # biome check . (lint + format check)
bun run lint:fix          # biome check --write . (auto-fix)
bun run format            # biome format --write .
bun test                  # run all tests
bun test --watch          # run tests in watch mode
bun run ci                # full pipeline: typecheck ‚Üí lint ‚Üí test ‚Üí build
```

## Testing Instructions

- Framework: `bun:test` (built into Bun, no extra config)
- Test files: `test/*.test.ts`
- Import source directly from `../src/index.ts` (Bun resolves `.ts` natively)
- Use `describe`/`it`/`expect` from `bun:test`

## Code Style Guidelines

### API Design
- **Prefer function-based over classes** ‚Äî pure functions, composable utilities
- **Minimal API surface** ‚Äî expose only what's needed. Re-export from `src/index.ts`, keep internals in `src/internal/`
- **Progressive complexity** ‚Äî simple by default, powerful when needed

### Code Style
- **Clean, readable, pragmatic** ‚Äî avoid cleverness
- **Strong TypeScript, strict mode**

### Structure
- **Modular over monolithic** ‚Äî prefer `lib/` with focused files over a single `utils.ts`
- **ESM-first, flat hierarchy when possible**
- **Convention over configuration**

### Conventions
- **File naming**: `kebab-case.ts`, `camelCase` for functions, `PascalCase` for types
- **Imports**: use `.ts` extension in import paths, `import type` for type-only imports
- **Internal modules**: `src/internal/` ‚Äî mark with `@internal` JSDoc
- **Pure functions**: prefer over classes unless state is needed

## Build

```bash
bun run build    # tsdown ‚Üí dist/index.mjs + dist/index.d.mts + sourcemaps
```

## Versioning and Release

Uses [Changesets](https://github.com/changesets/changesets).

Create a changeset when modifying `src/` or `test/`:

```bash
bunx changeset
```

| Conventional commit type | Changeset bump |
|---|---|
| `feat` + `BREAKING CHANGE` | major |
| `feat` | minor |
| `fix`, `perf`, `refactor`, `docs`, `chore`, `style`, `test` | patch |

Release flow:
1. Changeset `.md` files committed alongside code
2. `bunx changeset version` ‚Äî bumps versions, updates CHANGELOG
3. Create `release/v*.*.*` branch and PR against `main`
4. Merge PR
5. `bun run build && bunx changeset publish` ‚Äî publishes to npm, creates tag + GitHub release

## Git Hooks (Lefthook)

- **pre-commit**: `biome check --staged --write --unsafe` + `tsc --noEmit` (staged files only)
- **pre-push**: `bun test` + `bun run build`

## Important Notes

- **Bun is the package manager** ‚Äî don't use npm/pnpm. Keep `bun.lock` in sync
- **Never commit automatically** ‚Äî stage, changeset, and present the message, but user runs the commit
- **`package.json` `private` field** ‚Äî defaults to `true`. Set `false` before publishing to npm
- **`dist/`** ‚Äî generated by `bun run build`, do not modify directly
- **`CHANGELOG.md`** ‚Äî auto-generated by changesets, do not edit manually
- **`tmp/`** ‚Äî gitignored, use for scratch files
